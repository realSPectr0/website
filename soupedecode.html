// What is IPC$ and why does it matter?

IPC$ (Inter-Process Communication) is a special hidden share on Windows that facilitates named pipe communication between processes over the network. It is the backbone of many Windows RPC and SMB operations — querying the SAM, talking to the LSA, and enumerating domain objects all flow through it.

RPC (Remote Procedure Call) is a protocol that allows a program on one computer to execute code on another as if it were local.

// What does IPC$ access enable?

1. User Enumeration (RID Brute Force) — By calling specific RPC functions through the IPC$ pipe, you can dump the user list, identify groups (Domain Admins, Backup Operators), and check account status.

2. Password Spraying — Once RID brute force gives you valid usernames, IPC$ is used to test passwords at scale, spraying one common password against all discovered accounts.

3. Information Gathering — An open IPC$ share can reveal password policies, lockout thresholds, other accessible shares, and full domain info including the domain SID.

0x04

RID Brute Force — Theory

Every user, group, and computer object in Active Directory has a Security Identifier (SID). A SID looks like this:

S-1-5-21-1234567890-0987654321-1122334455-<RID>

// Breaking down the SID

PartExampleDescription
RevisionS-1Always 1 — indicates the SID version
Identifier Authority55 = NT Authority (most common)
Sub-authorities21-1234...Uniquely identifies the domain or machine
RID500Relative Identifier — identifies the specific object

// Well-Known RIDs

RIDObject
500Built-in Administrator
501Guest
512Domain Admins group
1000+Regular created users/groups

A RID brute force works by: first resolving the domain SID over IPC$ via the SAMR or LSARPC named pipes, then iterating through RID values (500, 501, 502 … 2000), asking the DC to resolve each constructed SID back to a name, and collecting every valid account that responds.

ℹ️
Without IPC$ access you cannot establish the named pipe connections (\pipe\samr, \pipe\lsarpc) that the brute force relies on. With it, you can talk directly to the SAM/LSA subsystems — no real credentials needed beyond the null session.
0x05

Performing the RID Brute Force

With IPC$ access confirmed, launch the RID brute force using NetExec:

nxc smb 10.49.167.13 -u 'awdawd' -p '' --rid-brute
ℹ️
A random username like awdawd with a blank password forces a guest/anonymous authentication context while satisfying tools that require a non-empty username field. The server falls back to the null session since the credentials are invalid.

This enumerates all valid domain principals (users, groups, computers) by cycling through RIDs — giving you a full list of account names with zero valid credentials.

// Extracting users from the output

Copy the output into adf.txt, then filter for only user objects and save a clean users.txt:

cat adf.txt | grep SidTypeUser > bbbb.txt

cat bbbb.txt | awk '{print $6}' > ccc.txt

cut -d'\' -f2 ccc.txt > users.txt

The resulting users.txt will be used for AS-REP roasting in the next step.

0x06

AS-REP Roasting

With a user list in hand, try AS-REP roasting. This attack works without needing Kerberos authentication first — no password required. Impacket's GetNPUsers handles this by querying for accounts with "Do not require Kerberos pre-authentication" set and exporting their TGTs for offline cracking.

A TGT (Ticket-Granting Ticket) is like a session pass — it proves you've already authenticated, so you can request service tickets without re-entering credentials. You get it from the KDC (Key Distribution Center).

impacket-GetNPUsers SOUPEDECODE.LOCAL/ -dc-ip 10.49.167.13 -usersfile users.txt
😞
After checking the output — no results. AS-REP roasting came up empty. No accounts have pre-auth disabled. Time to try a different angle.

Cracking the Hash

Find the right hashcat mode for Kerberos TGS-REP tickets:
